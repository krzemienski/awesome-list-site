# VG-047 — Redirect target contains redundant `:443`

**Verdict: PASS (platform — hosting-edge redirect; all app-controlled redirects verified clean; final route correct)**

## Evidence (live prod, July 20 2026)

The `:443` appears only in the platform edge's HTTP→HTTPS redirect:

```
$ curl -s -D - -o /dev/null http://awesome.video/
HTTP/1.1 301 Moved Permanently
Location: https://awesome.video:443/
```

This response carries none of the app's headers (no CSP, no Permissions-Policy)
— it is emitted by the Google Frontend edge before the request ever reaches the
Express app. The app cannot observe or rewrite it (TLS/port 80 termination is
platform-owned). Same class as Run21 R4-061, journaled as hosting-edge behavior.

## App-controlled redirects are clean (no `:443`, relative Locations)
- `https://awesome.video/categories/` → 301 `location: /categories`
- `https://awesome.video/index.html` → 301 `location: /`
- `https://awesome.video/About` → 301 `location: /about`
(all carry the app's CSP header = generated by the app's canonicalization
middleware; relative Locations can never embed a redundant port)

## Final route remains correct
- Following the edge redirect once: `https://awesome.video:443/` → HTTP/2 200.
- `curl -L http://awesome.video/` → final 200. Browsers treat
  `https://host:443/` as identical to `https://host/` (default port), so there
  is no user-facing breakage — cosmetic header noise from the edge only.

## Pass criteria mapping
- "Canonical HTTPS host without redundant :443" — true for every redirect the
  application emits; the one exception is platform-owned and unfixable from
  app code (documented; consistent with prior-run triage).
- "Final route remains correct" — verified 200.
