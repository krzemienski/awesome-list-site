---
name: SEO title two-pass parity
description: Server og-middleware title templates are the source of truth; client SEOHead titles must mirror them exactly.
---

Rule: any page title emitted client-side via SEOHead must produce the exact same string as the server's og-middleware template for that route (brand-first for home, `${name} — Awesome Video` for taxonomy, `Page Not Found — Awesome Video` for soft-404s).

**Why:** Googlebot's crawl pass captures the initial HTML title and its render pass captures the hydrated DOM title. If they differ (even both being "valid"), SERP titles can flip-flop. A May 2026 audit found the client composing titles differently (extra " Resources", parent-category segments, brand-last ordering) and aligned client → server.

**How to apply:** when changing a title template on either side, change both, then verify with a two-pass diff: fetch raw HTML title vs. Playwright post-hydration `document.title` (decode HTML entities before comparing, or `&amp;` produces false diffs). SEOHead's `withBrand` is idempotent — passing a title that already contains "Awesome Video" gets no suffix appended.

**Count-in-title trap (July 2026):** taxonomy titles/descriptions are now generated by pure functions in `shared/seo-templates.ts` imported by BOTH og-middleware and the client pages, so parity is by construction — never re-duplicate a template string by hand. Keep titles COUNT-FREE (category titles carry no number); then title parity can't drift even if the two count sources momentarily disagree. The HOME title is the one exception that embeds a count: both sides MUST read the SAME flat deduped-tree array (`resources.length` from `/api/awesome-list` on the client / `getTreeCached().resources` on the server), NOT a filtered per-category sum, or the crawl and render passes can differ.
