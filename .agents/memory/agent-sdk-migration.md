# Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`) in this repl

Used to run the admin Researcher + Enrichment agents as a real multi-agent system (orchestrator + subagents + custom in-process MCP tools). Replaced direct `@anthropic-ai/sdk` Messages API usage in those two flows.

## Dependency reality (the painful part)
- Recent agent SDK (0.3.150+, incl. 0.3.201) **hard-peers `zod ^4.0.0` + `@anthropic-ai/sdk >=0.93.0` + `@modelcontextprotocol/sdk ^1.29.0`**. This app is zod v3 everywhere (drizzle-zod ^0.7.0, every schema/form/route).
- **Decision: keep zod v3** (Path A). Added `legacy-peer-deps=true` to `.npmrc` so installs ignore the zod-v4 peer. Bumped `@anthropic-ai/sdk` 0.37 → 0.110 (its usage in claudeService.ts/recommendations.ts is only `messages.create`/`content[0].text`/`usage.*` — stable, no breakage; build verified).
- **Why not zod v4:** full v3→v4 migration also forces drizzle-zod/drizzle-orm bumps + createInsertSchema rewrites across the whole app — huge blast radius for zero product gain.
- **Proven at runtime:** `tool()` custom in-process MCP tools accept **zod v3** schemas fine despite the ^4 peer. Path A is safe.

## Runtime facts (verified by spike in-container)
- `query()` spawns a **bundled ~251MB linux-x64 native `claude` binary** (`node_modules/@anthropic-ai/claude-agent-sdk-linux-x64/claude`) as a subprocess per call; a trivial call completes ~13s (ttft ~2.7s). Use `startup()`/WarmQuery to move spawn cost off the critical path if latency matters.
- Binary reads `ANTHROPIC_API_KEY` from env automatically.
- **Per-run endpoint/auth = `options.env`** (Record passed to the subprocess). Setting `options.env.ANTHROPIC_BASE_URL` genuinely redirects the endpoint (verified: base_url=example.com → 405, did NOT hit the real API). So custom base URL + `ANTHROPIC_AUTH_TOKEN` per job works via `options.env` — no global process.env mutation, real per-job isolation.
- **Per-run model = `options.model`** (e.g. "claude-haiku-4-5", or "haiku"/"sonnet" shorthands).

## Multi-agent + tools
- Subagents via `options.agents` map (AgentDefinition: description/prompt/tools/model). Orchestrator delegates through the built-in delegation tool; stream emits `system` events `subtype: task_started|task_updated|task_notification` carrying `subagent_type`, `task_id`, `description` — ideal for persisted logs + a comms graph. There is also a built-in `SendMessage` inter-agent tool.
- Custom tools: `tool(name, desc, zodShape, handler)` + `createSdkMcpServer({name,tools})`; pass via `options.mcpServers`; tool is exposed as `mcp__<server>__<tool>`.
- **Tool lockdown gotcha:** `permissionMode: "bypassPermissions"` AUTO-APPROVES ALL tools; `allowedTools` does NOT restrict in that mode (spike model ran `Bash` unprompted). To keep a server-side agent from touching Bash/Edit/Write/Cron/filesystem, use `disallowedTools` (or a `canUseTool` permission callback), not just the allowlist.
- SDKMessage stream types seen: `system` (init lists full toolset; also thinking_tokens + task_* events), `assistant`, `user` (tool_result), `result` (is_error, api_error_status, duration_ms, duration_api_ms, num_turns, result text, usage).

## WebSearch, budget-abort, subagent attribution (spike-verified 2026-07-06)
- **WebSearch works in-container.** Built-in harness tool `WebSearch` returns REAL current results (single-agent phase C answered SVT-AV1 correctly, incl. that the GitHub repos are archived). The harness first calls a `ToolSearch` discovery step then `WebSearch`, so give GENEROUS `maxTurns` (12+) — a low turn budget starves the search (phase A recorded 0 findings purely from turn exhaustion, not a WebSearch failure).
- **Counting searches:** `usage.server_tool_use.web_search_requests` stays **0** for the harness WebSearch (that field is only for Messages-API server_tool_use). To count searches, count `tool_use` blocks named `WebSearch` in the assistant stream. Harness WebSearch results arrive as tool_result in `user` messages, not as `web_search_tool_result` content blocks.
- **Budget/cancel abort:** call `abortController.abort()` from the STREAM CONSUMER (while iterating), NOT from inside a tool handler (doing so wedges the loop — it never terminates, verified). Abort-from-consumer DOES terminate the loop, and **persisted tool side-effects done before the abort survive** (phase B: `findings=["alpha"]` intact). BUT abort makes the `for await` **throw a generic `Error`** (not clean end, name is just "Error"). So the wrapper MUST wrap iteration in try/catch and treat a throw as a controlled stop when `abortController.signal.aborted === true` (finalize + emit run_completed with reason budget/cancelled; do NOT rethrow).
- **Graceful cancel alternative:** `Query.interrupt(): Promise<void>` (Query extends AsyncGenerator) ends the loop via the SDK's own path — cleaner than abort for user-initiated cancel; hold the `q` reference to call it.
- **Subagent attribution (for agent_events + comms graph):** `system` `task_started`/`task_progress`/`task_notification` carry `subagent_type`+`task_id`(+usage on progress/notification). `assistant` messages carry `subagent_type` (e.g. "scout" vs undefined=orchestrator), `parent_tool_use_id`, and `message.usage` (input_tokens/output_tokens) → attribute tokens per actor. `total_cost_usd` on the `result` message is authoritative (phase A $0.075, phase C $0.04) → drop the hand-rolled pricing table.
- **Decomposition decision:** orchestrator (custom MCP tools + taxonomy prompt) + ONE scout subagent (WebSearch only). More subagents add cost/latency with no judgment gain (dedupe is deterministic Set lookup; categorization is already in the prompt).
- **Native budget/turn caps — do NOT hand-roll.** `Options.maxBudgetUsd` (number) + `Options.maxTurns` are enforced by the SDK: run stops with result `subtype: 'error_max_budget_usd'` / `'error_max_turns'`. So drop the pricing table AND the consumer-side budget-abort loop entirely; just pass both options and read `result.total_cost_usd`. Keep the `abortController` ONLY for user cancel (activeJobs map) — abort from consumer throws generic Error (catch + check `signal.aborted`).
- **Isolation options for server-side agents:** `settingSources: []` (don't load the repl's own CLAUDE.md/skills/settings), custom `systemPrompt` as a plain string (replaces the harness coding-assistant preamble), and `disallowedTools` baseline to block side-effecting/filesystem tools. Keep `Task`+`TaskCreate/Get/List/Output/Stop/Update` (delegation), `WebSearch`, `ToolSearch` (harness discovers WebSearch through it), `SendMessage`, `ReportFindings`, and `mcp__*`; disallow Bash/Edit/Write/NotebookEdit/Read/WebFetch/Cron*/DesignSync/*Worktree/Monitor/PushNotification/ScheduleWakeup/Workflow/Skill.

## Runtime-validated behavior (Researcher, real jobs 2026-07-06)
- **Native cap = result subtype THEN a throw.** On `maxBudgetUsd`/`maxTurns` the SDK emits a `result` message with `subtype: error_max_budget_usd`/`error_max_turns` (cost/turns captured) **and then the generator throws** `Error: Claude Code returned an error result: Reached maximum budget (...)`. The wrapper catch MUST treat "already have a result subtype OR message matches /maximum budget|max turns/" as a **controlled stop → completed**, not a failure: do NOT emit an error event, do NOT rethrow. (Verified: old code rethrew → job `failed`; fixed code → job `completed`, 0 error events, cost persisted.)
- **Bound the scout or it eats the whole budget.** An unbounded WebSearch subagent will run 10+ searches in ONE delegation and never return control, so the orchestrator never gets to `save_discovery` (whole budget spent, 0 saved). Fix is prompt-side: hard-cap "AT MOST 3-4 searches then STOP and report" in the scout prompt. After the cap, a real loop saves 10 discoveries across multiple gaps at ~$0.4.
- **`result.num_turns` ≠ `options.maxTurns`.** num_turns aggregates sub-agent model turns too, so a run can report 35 turns under maxTurns:20 and still end `subtype:success` (natural finish). maxTurns IS wired (options.maxTurns); don't "fix" the discrepancy.
- **In-process jobs die on server restart** → stuck at status `processing` (orphaned). The boot orphan-watchdog marks stuck jobs failed on next start — that's the safety net. To validate: run via HTTP on the persistent workflow (login `/api/auth/local/login` as admin with ADMIN_PASSWORD → cookie → POST `/api/researcher/start`), and track the tsx server PID (`pgrep -f "tsx/dist/loader.*server/index.ts"`) across the run to detect an infra restart that would silently orphan the job.

## Runtime-validated behavior (Enrichment, real jobs 2026-07-06)
- **Per-job counter bumps MUST be SQL-atomic.** The enrichment orchestrator's `submit_enrichment` MCP tool can fire several times **concurrently in one assistant turn**, so a read-job-then-write-`+1` counter update loses increments and drops ids from the jsonb id arrays. Use DB-side `COALESCE(col,0)+1` and jsonb `COALESCE(col,'[]'::jsonb) || '[id]'::jsonb`. (research jobs never hit this because saves came across separate turns — but the correct idiom is the SQL increment, mirrored from researchService.) The jsonb id columns are NOT pg arrays → concat with `||`, not `array_append`.
- **User-cancel must short-circuit the retry/stall loop.** The one-run-per-batch loop bumps `retryCount` on every queue item still `pending` after a batch run (stall guard, force-fail at maxRetries). On abort, check `abortController.signal.aborted` and `break` BEFORE that retry loop, else a user cancel counts as a retry and can force-fail an item that was at maxRetries-1.
- **Enrichment = single orchestrator, NO subagent.** Two in-process MCP tools only: `get_pending_batch` (returns a pre-scraped batch — scraping is done server-side in prepareBatch, not by the agent) + `submit_enrichment` (deterministic persist). `extraDisallowedTools` locks out `Task*`/`WebSearch`/`SendMessage` on top of the baseline. Cost/turns/agentLog live in `enrichment_jobs.metadata` jsonb (no cost/token columns on that table — no migration).

## Per-run custom-endpoint security invariant (2026-07-06, architect-flagged)
- **A custom `baseUrl` must NEVER ride the platform `ANTHROPIC_API_KEY`.** `buildAgentEnv` deletes `ANTHROPIC_API_KEY` whenever `config.baseUrl` is set (not only when a token is present), and `parseAgentConfigFromRequest` rejects `baseUrl`-without-`authToken` with a user-facing 400. **Why:** the platform key is only valid against the platform host; the original code dropped the key only when a token was present, so a blank-token + custom-baseUrl run would ship the platform key as credentials to an arbitrary https host (SSRF guard only blocks private IPs, not public attacker hosts). **How to apply:** any future per-run endpoint override must keep BOTH guards — key-drop keyed on baseUrl, and token-required-with-baseUrl — plus keep both UI hint strings honest ("required if a base URL is set").
- **DNS-rebinding TOCTOU:** `validateBaseUrl` is re-run at run start inside `runAgentQuery` (not just at job creation) so a host that rebinds to a private IP after validation fails closed before the endpoint reaches the SDK subprocess.

## Non-interactive hygiene
- Always run non-interactively (server/scripts): set an `abortController` + timeout; a bad endpoint otherwise hangs on ret/backoff with no output.
