# BUG-004 / BUG-035 fix

## root cause
The site uses a client-side React SPA. Before hydration, the HTML shell is empty
(`<div id="root"><!--app-html--></div>`). The React component `AppHeader.tsx`
renders the search button + sidebar trigger after hydration. Non-JS clients
(crawlers, accessibility tooling) saw zero inputs and no nav affordance on the
landing page. Same root cause as BUG-042/044.

## files changed
- `server/og-middleware.ts`
  - Added `homeShellChrome()` helper that emits `<header>` with a hamburger
    `<button aria-label="Toggle navigation menu" data-testid="mobile-drawer-trigger">`
    and a `<form action="/search" method="get" role="search">` containing a real
    `<input type="search" name="q">`.
  - Prepended `homeShellChrome()` to `bodyHtml` in the home route branch.

## before
`GET /` HTML (curl, no JS) had:
- 0 `<input>` elements
- no `<button aria-label="Toggle navigation menu">`
- no `<form action="/search">`
- only an empty `<div id="root"><!--app-html--></div>` shell

## after
`GET /` HTML (curl, no JS) has:
- 1 `<input type="search" name="q" placeholder="Search resources…">`
- 1 `<button aria-label="Toggle navigation menu">`
- `<form action="/search" method="get" role="search">`
- `<header data-testid="ssr-home-header">…</header>`
- search button + hamburger rendered as real form/button so password managers
  and crawlers see the affordances

The client React `AppHeader` still hydrates on top, maintaining interactive
behavior. The static HTML is the no-JS fallback for the same affordances.
